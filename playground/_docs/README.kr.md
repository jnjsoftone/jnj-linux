# 📚 프로젝트 문서 허브

**${PROJECT_NAME}** 문서에 오신 것을 환영합니다. 이 `_docs/` 폴더는 이해관계자, 디자이너, 개발자, 운영팀 간의 지속적인 협업을 지원하도록 설계된 모든 프로젝트 정보의 단일 진실 공급원(Single Source of Truth)입니다.

## 📌 문서 범위

**이 문서는 프로젝트 레벨 문서입니다** - 애자일 개발 중 지속적으로 발전하는 프로젝트별 문서 템플릿입니다.

**주요 특징**:
- **프로젝트 전용**: 이 프로젝트만의 고유한 구현 세부사항
- **애자일**: 개발 생명주기 전반에 걸쳐 지속적으로 업데이트
- **AI 지원**: Claude Code가 코드 변경과 함께 자동으로 문서 업데이트
- **템플릿 기반**: 구조를 제공하며, 각 프로젝트가 구체적인 내용을 채움
- **이중 언어**: 영문(`.md`)은 AI 소통용, 한글(`.kr.md`)은 개발자용

**플랫폼 문서와의 관계**:
- 플랫폼 문서 (`docker-ubuntu/_docs/`) = **플랫폼 전체 표준** (코딩 규칙, 아키텍처 패턴)
- 프로젝트 문서 (여기) = **프로젝트 구현 세부사항** (요구사항, 기능, API 명세)
- 전체 문서 아키텍처는 [CLAUDE.md](/volume1/homes/jungsam/dockers/CLAUDE.md) 참조

**AI 문서 업데이트**:
- 코드 변경 시, Claude Code가 **자동으로 식별**하여 업데이트가 필요한 문서 찾음
- 업데이트는 코드 변경 후가 아닌 **동시에** 진행됨
- 영문과 한글 버전이 **동기화 유지**됨
- 세부 프로세스는 아래 "문서 업데이트 워크플로우" 섹션 참조

## 🎯 문서화 철학

이 문서는 다음을 따릅니다:
- **살아있는 문서(Living Documentation)**: 코드 변경과 함께 지속적으로 업데이트
- **워크플로우 중심**: 개발 생명주기에 맞춰 구성
- **협업 우선**: 모든 팀원 간 소통 촉진
- **버전 관리**: Git으로 모든 문서 추적 및 변경 이력 관리
- **AI 유지관리**: Claude Code가 문서를 최신 상태로 정확하게 유지

## 🔄 문서 업데이트 워크플로우

### AI 지원 문서 업데이트 (Claude Code)

**자동 업데이트 프로세스**:

```
코드 변경 → Claude 영향 탐지 → 영향받는 문서 식별 → 내용 업데이트 → 동기화 검증
      ↓            ↓                     ↓                    ↓              ↓
기능 구현    변경사항 분석      요구사항/API/개발      .md 업데이트      .kr.md 확인
아키텍처    CLAUDE.md 참조     아키텍처/테스팅     예제 추가        양쪽 동기화
API 변경    양쪽 _docs 확인    배포/운영           참조 업데이트     유효성 검사
```

**Claude Code가 업데이트하는 내용**:

1. **기능 구현**:
   - `03-planning/01-roadmap.md` - 기능을 완료로 표시
   - `05-api/graphql/*.md` 또는 `05-api/rest/*.md` - 새 엔드포인트 추가
   - `06-development/05-implementation-notes/{feature}.md` - 구현 내용 문서화
   - `98-chats/claude/{date}-{topic}.md` - 개발 결정사항 기록

2. **아키텍처 결정**:
   - `04-architecture/decisions/ADR-{번호}-{제목}.md` - 아키텍처 결정 기록(ADR) 추가
   - `04-architecture/diagrams/` - 다이어그램 업데이트 (필요시)
   - 플랫폼 문서 (재사용 가능한 패턴인 경우)

3. **API 계약 변경**:
   - `05-api/graphql/*.md` 또는 `05-api/rest/*.md` - 스키마/엔드포인트 업데이트
   - `05-api/examples/*.md` - 실제 작동하는 코드로 예제 업데이트
   - `06-development/05-implementation-notes/*.md` - 주요 변경사항 메모

4. **테스팅 & QA**:
   - `07-testing/02-test-cases/*.md` - 테스트 시나리오 추가
   - `07-testing/03-automated-tests/*.md` - 테스트 커버리지 문서화

5. **배포 변경**:
   - `08-deployment/01-deployment-guide.md` - 절차 업데이트
   - `08-deployment/04-changelog.md` - 버전 히스토리 추가

**Claude Code의 체크리스트** (자동 검증):
- ✅ 영향받는 파일 식별 및 업데이트 완료
- ✅ 영문과 한글 버전 동기화 완료
- ✅ 줄 번호와 함께 파일 경로 명시 (예: `api/src/users.ts:42`)
- ✅ 코드 예제 테스트 및 작동 확인
- ✅ 문서 간 상호 참조 유효성 확인
- ✅ 기존 문서 구조 준수

### 문서 업데이트 시점

```mermaid
코드 변경 → 기술 문서 업데이트 → PR 리뷰 → 병합 → 배포
    ↓              ↓                      ↓           ↓
요구사항      API 문서              코드 리뷰     체인지로그
 변경       아키텍처           문서 확인        릴리스 노트
          개발 가이드
```

### 수동 문서 검토 주기

- **일일**: API 문서, 개발 가이드 (Claude Code 지원)
- **주간**: 사용자 스토리, 스프린트 계획, 상태 보고서
- **스프린트**: 로드맵 업데이트, 기능 명세
- **분기**: 아키텍처 검토, 기술 결정
- **연간**: 디자인 시스템, 전체 전략

### 사람의 책임

Claude Code가 기술 문서 업데이트를 처리하는 동안, **사람은 다음을 담당합니다**:
- PR에서 문서 변경 사항 검토 및 승인
- 비즈니스 요구사항 및 목표 업데이트 (`01-requirements/`)
- 디자인 명세 작성 (`02-design/`)
- 로드맵 및 사용자 스토리 계획 (`03-planning/`)
- 상태 보고서 및 회의록 작성 (`10-collaboration/`)
- 최종 아키텍처 결정 내리기 (Claude는 이를 문서화함)
